Section 02

Executive Summary

Vision Statement

The Autonomous AI Job Application Platform is an intelligent career agent system that continuously discovers, evaluates, tailors, applies, and learns — acting as a tireless professional advocate for its user. Built for the modern AI-first job market, it transforms what is currently an exhausting, manual, and largely ineffective process into a precision-engineered pipeline that delivers measurable outcomes: more interviews, higher offer rates, and dramatically less time spent on administrative job-search mechanics.

This platform is not a simple automation script. It is a sophisticated multi-agent orchestration system combining state-of-the-art LLMs, semantic search, browser automation, and continuous reinforcement learning into a coherent, auditable, and responsible product. Every action is grounded in evidence from the user's verified portfolio, every claim is traceable to a source of record, and every submission meets a quantifiable quality bar before it leaves the system.

Problem Statement

Job searching at a senior technical level is paradoxically inefficient. A candidate of Vikram Deshpande's calibre — 15+ years of delivery leadership, AI/ML architecture, and enterprise program management across ATO, ANZ, NAB, MYOB, Microsoft, and Telstra — should be spending time on high-value activities: preparing for interviews, networking strategically, and evaluating opportunities. Instead, the reality is:

  • Manually tailoring resumes for each application takes 45-90 minutes per submission
  • ATS systems reject up to 75% of qualified candidates for keyword mismatch rather than capability gaps
  • Cover letter generation is repetitive, yet each must be evidence-specific and company-aware
  • Tracking applications across LinkedIn, SEEK, Indeed, and direct portals requires constant context switching
  • Recruiter follow-up and calendar management create an additional cognitive load
  • Without analytics, there is no feedback loop to improve the application strategy over time

Solution Overview

The platform addresses every problem in the job-search lifecycle through a modular multi-agent architecture. The system operates as follows: a Job Discovery Agent continuously monitors job boards and APIs; a Planner Agent evaluates fit against the user's profile; a Resume Tailoring Agent produces ATS-optimised resumes without changing a single pixel of the source formatting; a Cover Letter Agent generates evidence-backed, company-specific letters; an Application Agent submits the package via browser automation with human approval gates; and a Learning Agent continuously refines the strategy based on outcomes. Every step is monitored by a Compliance Agent that guarantees no fabrication, no PII leakage, and full audit trail.

Business Goals

  • Reduce active job-search time from 20+ hours per week to fewer than 4 hours
  • Increase interview conversion rate from typical 2-3% to 8-12% (3x improvement)
  • Submit 5-10 high-quality applications per day versus 1-2 manually
  • Achieve 85%+ ATS pass rate for every submitted resume
  • Build a continuously improving application strategy that compounds over time
  • Maintain 100% factual accuracy — zero fabricated achievements or skills

Architecture Principles

Separation of Concerns

Each agent has a single, well-defined responsibility. The Supervisor orchestrates; it does not execute. The Resume Tailoring Agent tailors; it does not submit. This prevents tight coupling and enables independent scaling and testing of each agent.

Event-Driven Messaging

All inter-agent communication flows through the Redis message queue. Agents subscribe to task queues and publish results as events. This decouples producers from consumers, enables async processing, and provides natural dead-letter queue support for failure handling.

Immutable Audit Trail

Every agent action — read, write, decision, or escalation — is logged to an immutable audit store. This enables full forensic replay of any application history, supports compliance verification, and provides the training data for the Learning Agent.

Section 05

AI Agent Architecture

The platform comprises 20 specialised AI agents, each with a clearly defined responsibility, input/output contract, tool access, memory scope, decision logic, and failure handling strategy. Agents communicate asynchronously via the Redis message queue and share state through the PostgreSQL knowledge base. All agents are implemented as LangGraph nodes within a directed acyclic graph workflow, enabling parallel execution, conditional branching, and human-in-the-loop gates.

Agent Design Principles: Single Responsibility — one agent, one function. Verifiable Outputs — every agent produces a confidence score alongside its output. Human Escalation — any confidence below threshold triggers an approval gate. Idempotent Execution — agents can be retried safely without side effects. Full Audit — every invocation is logged with inputs, outputs, timestamps, and reasoning.

Core Execution Agents (1-7)

01
Job Discovery Agent
Continuous job board surveillance and ranking
Responsibilities Crawls job boards (LinkedIn, SEEK, Indeed), RSS feeds, and company careers APIs on a configurable schedule. De-duplicates listings. Scores relevance against user profile.
Inputs Search criteria, keyword filters, salary range, location, seniority level, excluded companies
Outputs Ranked job list with relevance score, JD text, company metadata, application URL, posting date
Tools Playwright (browser), LinkedIn API, SEEK API, Indeed RSS, web scraper
Memory Scope Long-term: previously seen jobs, applied companies. Short-term: current crawl session
Decision Logic Relevance score > 70%: queue for evaluation. 50-70%: human review. <50%: discard with reason logged.
Failure Handling Retry with exponential backoff (3 attempts). If API rate-limited, switch to RSS fallback. Alert on consecutive failures exceeding 5.
02
Resume Tailoring Agent
JD-specific resume optimisation with format preservation
Responsibilities Tailors resume content to match the job description using semantic skill mapping and keyword insertion. Preserves exact source formatting — typography, spacing, columns, and margins unchanged.
Inputs Base resume (PDF), job description text, portfolio evidence library, keyword priority list
Outputs Tailored resume PDF, ATS score, diff report showing changes, confidence score
Tools PyMuPDF (parser), python-docx, LLM (GPT-4o), ATS Scorer, python-docx formatter
Decision Logic Only proceed to submission if ATS score > 75%. Below threshold: escalate to human with gap report.

Section 06

Multi-Agent Orchestration

The orchestration layer is the nervous system of the platform. It coordinates the 20 agents through a Supervisor → Planner → Executor → Reviewer → Verifier hierarchy, using LangGraph to define the execution DAG and Redis BullMQ as the asynchronous task bus. Coordination Protocol

Task Lifecycle

Tasks are published to typed queues (job-discovery, resume-tailor, etc.) by the Planner. Agents consume tasks, update shared state via the LangGraph StateGraph, and publish completion events. Downstream agents subscribe to relevant completion events to trigger their own execution.

Conflict Resolution

The Supervisor Agent maintains a global goal registry. When two agents produce conflicting outputs (e.g., different ATS scores for the same resume), the Supervisor invokes a tiebreaker: prefer the higher-confidence result, flag the discrepancy, and log both for audit. Persistent conflicts escalate to human.

Human-in-the-Loop Gates

Four mandatory escalation points: (1) Job approval — user confirms which discovered jobs to pursue. (2) Application approval — user approves tailored resume + cover letter bundle before submission. Parallel Execution Model

Once a job is approved for application, the Portfolio Agent, ATS Optimization Agent, and Recruiter Intelligence Agent execute in parallel. Resume Tailoring and Cover Letter generation are serialized (Cover Letter depends on tailored resume). This hybrid model achieves maximum throughput while respecting data dependencies.

State Management

LangGraph StateGraph maintains a typed state object per job application instance. State includes: current stage, all agent outputs, human decisions, error counts, and audit trail. State is persisted to PostgreSQL with every transition — crash recovery always resumes from the last committed state checkpoint.

Keyword Extraction Algorithm

The keyword extraction process uses a three-stage pipeline applied to each job description:

  1. TF-IDF Analysis: Compute term frequency-inverse document frequency across a corpus of 10,000+ job descriptions to identify high-signal keywords. Common filler terms are filtered out; role-specific technical terms are upweighted.
  2. Named Entity Recognition: spaCy NER pipeline identifies technologies (PRODUCT), organizations (ORG), and skills (custom entity type). These receive higher priority in matching regardless of TF-IDF score.
  3. Dependency Parsing: Identify requirement phrases (e.g., "must have 5+ years of Python") to extract structured requirement slots: skill, level, duration, and importance qualifier (required/preferred).

The skill gap detector classifies each JD requirement into one of four states:

Security Architecture

The platform handles sensitive personal and professional data including resumes, employment history, contact details, credentials, and application communications. Security is implemented using a Zero Trust model — every request is authenticated, every action is authorised, every data access is logged, and no component implicitly trusts any other. The architecture is designed for compliance with the Australian Privacy Act 1988 and GDPR.

Authentication & Authorisation

OAuth2 + JWT Authentication

All user-facing APIs require a valid JWT Bearer token obtained via OAuth2 PKCE flow (NextAuth.js). Tokens expire after 1 hour. Refresh tokens are rotated on every use and stored in HttpOnly cookies. API-to-API communication between services uses short-lived machine tokens (15-minute TTL).

Role-Based Access Control (RBAC)

Three roles: Owner (full access, can modify profile and approve submissions), Reviewer (read-only access to applications and reports, cannot approve submissions), and System (internal service accounts with scoped permissions per agent type). Permissions are checked on every API call via middleware — never assumed.

  • Format Corruption Recovery: Revert to base resume, re-run tailoring from Stage 4 with stricter format constraints. Log the violation pattern for ATS Engine improvement.
  • LLM Provider Outage: Automatic failover — OpenAI GPT-4o → Claude 3.5 Sonnet → Gemini Pro. All models produce identical output contracts via standardised prompts.
  • Database Connection Lost: Write queue buffers mutations. On reconnection, replay mutations in order. Read requests serve from Redis cache with staleness flag displayed to user.
